# Local RAG Engine

A production-style Retrieval-Augmented Generation (RAG) API built with FastAPI, LangChain, ChromaDB, and local Ollama models.

This project lets you upload PDF or text documents, convert them into searchable vector embeddings, store them locally in ChromaDB, and ask questions against the uploaded knowledge base. The answer is generated by a local Ollama LLM using only the relevant document context retrieved from the vector database.

The goal of this project is simple: build a clean, understandable, local-first RAG backend that can be used as a foundation for real applications.

---

## What This Project Does

This application provides a complete document question-answering backend:

- Upload `.pdf` and `.txt` files through a FastAPI endpoint
- Load documents using LangChain document loaders
- Split large documents into smaller overlapping chunks
- Generate local embeddings using Ollama
- Store embeddings in a persistent ChromaDB vector store
- Retrieve the most relevant chunks for a user question
- Send retrieved context to a local Ollama LLM
- Return an answer with source file references

Everything runs locally. No external LLM API key is required.

---

## Why RAG?

Large Language Models are powerful, but they do not automatically know the contents of your private documents. They can also guess when they do not have enough information.

RAG solves this by connecting the LLM to your own data:

1. Documents are loaded and split into chunks.
2. Each chunk is converted into a vector embedding.
3. Embeddings are stored in a vector database.
4. When a question is asked, the most relevant chunks are retrieved.
5. The LLM receives only those chunks as context.
6. The LLM answers based on the retrieved context.

This makes answers more grounded, transparent, and useful for document-based applications.

## How The System Works

### 1. Document Upload

A user uploads a PDF or text file through:

```text
POST /api/v1/documents/upload
```

The API saves the file inside the `data/` folder.

### 2. Document Loading

The RAG pipeline loads the document:

- PDFs are loaded with `PyPDFLoader`
- Text files are loaded with `TextLoader`

### 3. Chunking

The document is split using `RecursiveCharacterTextSplitter`.

Default settings:

| Setting | Value |
|---|---:|
| Chunk size | `500` |
| Chunk overlap | `50` |

Chunking keeps the retrieval process accurate and prevents the LLM prompt from becoming too large.

### 4. Embedding

Each chunk is converted into an embedding using the local Ollama embedding model:

```text
nomic-embed-text
```

### 5. Vector Storage

The embeddings are stored in ChromaDB under the `vectorstore/` directory.

The API supports collection names, so you can separate different document groups.

### 6. Retrieval

When a user asks a question, the app retrieves the top matching chunks from ChromaDB.

Default retrieval count:

```text
RETRIEVER_K=3
```

### 7. Answer Generation

The retrieved chunks are passed into the prompt along with the user's question.

The local Ollama LLM generates the final answer using:

```text
llama3.2:latest
```

The prompt is designed to answer only from the retrieved context. If the answer is not available in the document, the model should say that it does not have enough information.

## Prerequisites

Install the following before running the project:

- Python 3.10 or higher
- Ollama installed locally
- At least one local LLM model pulled in Ollama
- The embedding model pulled in Ollama

Check that Ollama is running:

```bash
ollama --version
```

Pull the required models:

```bash
ollama pull llama3.2
ollama pull nomic-embed-text
```

Ollama should be available at:

```text
http://localhost:11434
```

## Environment Configuration

The application reads settings from `app/config.py` and can also load overrides from `.env`.

Example `.env`:

```env
APP_NAME=local-rag-engine
APP_VERSION=1.0.0
DEBUG=true

DATA_DIR=data
VECTORSTORE_DIR=vectorstore

CHUNK_SIZE=500
CHUNK_OVERLAP=50
RETRIEVER_K=3

LLM_MODEL=llama3.2:latest
EMBED_MODEL=nomic-embed-text
OLLAMA_BASE_URL=http://localhost:11434
```

Current default configuration:

| Variable | Default | Description |
|---|---|---|
| `APP_NAME` | `local-rag-engine` | API application name |
| `APP_VERSION` | `1.0.0` | API version |
| `DATA_DIR` | `data` | Folder for uploaded documents |
| `VECTORSTORE_DIR` | `vectorstore` | Folder for ChromaDB persistence |
| `CHUNK_SIZE` | `500` | Maximum chunk size |
| `CHUNK_OVERLAP` | `50` | Overlap between chunks |
| `RETRIEVER_K` | `3` | Number of chunks retrieved per question |
| `LLM_MODEL` | `llama3.2:latest` | Ollama model used for answers |
| `EMBED_MODEL` | `nomic-embed-text` | Ollama model used for embeddings |
| `OLLAMA_BASE_URL` | `http://localhost:11434` | Local Ollama server URL |

### Upload A Document

```http
POST /api/v1/documents/upload
```

Supported file types:

- `.pdf`
- `.txt`

Form fields:

| Field | Type | Required | Description |
|---|---|---|---|
| `file` | File | Yes | PDF or text file to ingest |
| `collection_name` | String | No | ChromaDB collection name. Defaults to `default` |

Example using curl:

```bash
curl -X POST "http://127.0.0.1:8000/api/v1/documents/upload?collection_name=my_docs" \
  -F "file=@data/sample.pdf"
```

Example response:

```json
{
  "filename": "sample.pdf",
  "chunks_created": 12,
  "status": "success",
  "message": "Successfully processed sample.pdf"
}
```

---

### Ask A Question

```http
POST /api/v1/chat/ask
```

Request body:

```json
{
  "question": "What is this document about?",
  "collection_name": "my_docs"
}
```

Example using curl:

```bash
curl -X POST "http://127.0.0.1:8000/api/v1/chat/ask" \
  -H "Content-Type: application/json" \
  -d "{\"question\":\"What is this document about?\",\"collection_name\":\"my_docs\"}"
```

Example response:

```json
{
  "question": "What is this document about?",
  "answer": "The document explains ...",
  "sources": [
    "data/sample.pdf"
  ],
  "status": "success"
}
```

## Recommended Workflow

1. Start Ollama.
2. Pull the required models.
3. Create and activate the Python virtual environment.
4. Install dependencies.
5. Start the FastAPI server.
6. Open `http://127.0.0.1:8000/docs`.
7. Upload a PDF or text file.
8. Ask questions using the same `collection_name`.

For example:

```text
Upload document to collection: my_docs
Ask question from collection: my_docs
```

Using the same collection name is important because the query endpoint searches inside that collection.

## Troubleshooting

### Ollama model not found

Pull the required model:

```bash
ollama pull llama3.2
ollama pull nomic-embed-text
```

### Ollama connection error

Make sure Ollama is running locally:

```bash
ollama serve
```

Then confirm the server is available at:

```text
http://localhost:11434
```

### Upload fails

Check that the file extension is supported. The API currently accepts only:

```text
.pdf
.txt
```

### Question returns weak or incomplete answers

Try these improvements:

- Upload a more relevant document
- Ask a more specific question
- Increase `RETRIEVER_K`
- Tune `CHUNK_SIZE` and `CHUNK_OVERLAP`
- Use a stronger Ollama LLM model if your machine supports it

### Answers are not using the expected document

Make sure the upload and chat request use the same `collection_name`.

## Summary

This is a local-first RAG backend for asking questions over your own documents. It combines FastAPI for the API layer, LangChain for the RAG pipeline, ChromaDB for vector search, and Ollama for local language and embedding models.

It is simple enough to learn from, but structured in a way that can grow into a more complete document intelligence system.
